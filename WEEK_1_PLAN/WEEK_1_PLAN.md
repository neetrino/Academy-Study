# День 1 — Правильные промпты: ТЗ → план → код

## Цель

Научиться писать задания AI как инженер, а не как клиент в комментариях.

---

## Теория (изучить перед практикой)

| Тема | Суть |
|------|------|
| Структура промпта | Контекст → цель → ограничения → критерии → шаги → риски |
| Плохой промпт | «Сделай красиво», «Пофикси», «Добавь функцию» — без критериев |
| Хороший промпт | Ясная цель, конкретные ограничения, критерии проверки, ожидаемый результат |

**Формула:**

```
[Контекст] Что уже есть, какой проект
[Цель] Что нужно сделать
[Ограничения] Стек, стиль, что нельзя трогать
[Критерии] Как понять, что готово
[Шаги] Если нужно пошагово
[Риски] Что может сломаться
```

---

## Практика — чеклист

- [ ] Выбрать одну задачу (рефактор / багфикс / фича)
- [ ] Написать **10 промптов** на неё: от плохого («сделай») к хорошему (полная структура)
- [ ] Сравнить ответы AI: разница в качестве, конкретности, полноте
- [ ] Зафиксировать лучший промпт

---

## Критерии проверки (день засчитан, если)

- [ ] AI выдаёт **план** + уточнения + предложенный diff, а не «вот код наугад»
- [ ] Есть **шаблон промпта** в `docs/PROMPT_TEMPLATE.md`

---

## Типичные ошибки (избегать)

- «Сделай красиво» без критериев
- Не указывать стек/ограничения — AI угадывает и ошибается

---

## Домашнее задание (День 1)

- [ ] Создать шаблон **`docs/PROMPT_TEMPLATE.md`** по структуре: контекст → цель → ограничения → критерии
- [ ] Добавить 2–3 примера: плохой промпт vs хороший

# День 2 — Chat vs Agent/Composer мышление

## Цель

Выбирать правильный режим под задачу.

---

## Теория (изучить перед практикой)

| Режим | Когда использовать | Что умеет |
|-------|-------------------|-----------|
| **Chat** | Объяснить, ответить, сгенерировать фрагмент | Текст, код в ответе, без изменений файлов |
| **Agent / Composer** | Изменить 2+ файлов, рефакторинг, фича | Редактирование файлов, команды в терминале, работа по проекту |

**Ссылка:** [Agent Overview \| Cursor Docs](https://cursor.com/docs/agent/overview)

---

## Практика — чеклист

- [ ] **3 задачи «только объяснить»** — Chat: «объясни этот код», «какая разница между X и Y», «как работает Z»
- [ ] **3 задачи «изменить 3–5 файлов»** — Agent/Composer: добавить компонент, рефакторинг модуля, подключить API
- [ ] Сравнить: когда Chat достаточно, когда нужен Agent

---

## Критерии проверки (день засчитан, если)

- [ ] Стажёр умеет формулировать задачу на изменение нескольких файлов
- [ ] Умеет принимать diff **осознанно** (читать, проверять, не принимать слепо)
- [ ] Есть **чек-лист выбора режима** в `docs/`

---

## Типичные ошибки (избегать)

- Пытаться «болтать» в чате, когда нужно править код — Agent быстрее и точнее

---

## Домашнее задание (День 2)

- [ ] Создать **чек-лист выбора режима** (Chat vs Agent) в `docs/CHAT_VS_AGENT_CHECKLIST.md`
- [ ] Добавить 3–5 вопросов для самопроверки перед выбором режима

# День 3 — Контекст: что давать AI, а что не надо

## Цель

Точечно кормить AI нужными файлами. Лишний контекст ухудшает ответы.

---

## Теория (изучить перед практикой)

| Понятие | Суть |
|---------|------|
| Авто-контекст | Cursor сам подставляет открытые файлы и т.п. |
| Ручной контекст | @упоминание файлов, папок, правил |
| «Всё сразу» | Слишком много контекста — качество падает, AI «теряется» |

**Ссылка:** [Context \| Cursor Learn](https://cursor.com/learn/context)

---

## Практика — чеклист

- [ ] Одна и та же задача с **разным контекстом**:
  1. Без контекста
  2. С 1 файлом
  3. С 3 файлами
  4. С правилами проекта (`.cursor/rules`)
- [ ] Сравнить качество ответов
- [ ] Определить минимально достаточный контекст для типовых задач

---

## Критерии проверки (день засчитан, если)

- [ ] Стажёр умеет **собрать минимальный контекст** для изменения
- [ ] Есть **контекст-пакеты** (наборы файлов) для 2–3 типовых задач

---

## Типичные ошибки (избегать)

- «Вот весь репозиторий, разбирайся» — перегрузка контекста

---

## Домашнее задание (День 3)

- [ ] Для каждого: какие файлы включать, какие не надо

# День 4 — Мини-рефактор с безопасной сеткой

## Цель

Делать улучшения без разрушений. Маленькие PR, атомарные изменения.

---

## Теория (изучить перед практикой)

| Принцип | Суть |
|---------|------|
| Маленькие PR | Один рефактор = один PR. Не мешать с фичами. |
| Атомарность | Каждое изменение можно откатить отдельно |
| Безопасная сетка | Линтер + тесты **до** и **после** — обязательны |

---

## Практика — чеклист

- [ ] Выбрать компонент/модуль для рефакторинга
- [ ] Определить scope: что меняем, что не трогаем
- [ ] Сделать рефакторинг через Agent
- [ ] Добавить/обновить тест или проверку
- [ ] Убедиться: линтер и тесты проходят
- [ ] Оформить PR (описание: что сделано, как проверить, риски)

---

## Критерии проверки (день засчитан, если)

- [ ] PR читается и **откатывается без слёз**
- [ ] Не смешаны рефакторинг и новая функциональность

---

## Типичные ошибки (избегать)

- «AI переписал половину проекта, но так надо» — контролировать scope

---

## Домашнее задание (День 4)

- [ ] Сделать **1 PR** в учебный репозиторий (рефакторинг + тест)
- [ ] Описание PR по формату: что сделано + как проверить + риски

# День 5 — Debugging через AI (без гадания)

## Цель

Уметь чинить баги через гипотезы. Фикс подкреплён логами/тестом.

---

## Теория (изучить перед практикой)

| Этап | Суть |
|------|------|
| Симптомы | Что именно сломалось: сообщение об ошибке, поведение |
| Гипотезы | Возможные причины (2–5 вариантов) |
| Проверка | Как проверить каждую гипотезу |
| Фикс | Минимальное изменение, устраняющее причину |
| Регресс-тест | Убедиться, что ничего не сломалось |

**Цикл:** симптомы → гипотезы → проверка → фикс → регресс-тест

---

## Практика — чеклист

- [ ] **3 бага** разного типа: UI, API, DB/логика
- [ ] Для каждого: сформулировать симптомы, выдвинуть гипотезы, проверить, зафиксировать
- [ ] Использовать AI для гипотез и предложений фикса — но **принимать решение самому**
- [ ] Задокументировать: Bug report + Fix rationale

---

## Критерии проверки (день засчитан, если)

- [ ] Фикс **подкреплён** логами или тестом, а не «вроде работает»
- [ ] Есть документ «Bug report + Fix rationale» для минимум 1 бага

---

## Типичные ошибки (избегать)

- «Пофиксил, не знаю почему» — всегда понимать причину

---

## Домашнее задание (День 5)

- [ ] Заполнить минимум 1 пример: описание бага, гипотезы, выбранный фикс, обоснование
