## Введение

> **Наконец-то.** Вместо «давай обучим стажёров вайбкодингу» вы хотите **нормальный учебный план**, а не шаманский танец вокруг кнопки *"Generate"*.

Ниже готовый **силлабус на 4 недели** (можно ужать в 2 недели, просто сливаете недели 1+2 и 3+4 и режете практику).

---

## Ключевая информация

| Параметр | Описание |
|----------|----------|
| **Цель** | Превратить выпускника базового курса программирования в разработчика, который умеет **управлять AI**, а не «просить кодик». |
| **Результат** | Стажёр делает фичи и фиксы в реальном репозитории, пишет тесты, делает PR, проходит ревью, умеет документировать решения, не ломает прод «потому что AI так сказал». |
| **Формат** | 4 недели, 5 дней в неделю, 2–4 часа/день (интенсив). |
| **Артефакты** | Мини-проект, набор промптов, `.cursor/rules`, чек-лист качества, README, PR-история. |

| Элемент | Описание |

# Неделя 1 — База AI-coding
*Навыки, без которых дальше только боль*

---

### День 1 — Правильные промпты: ТЗ → план → код

| Элемент | Описание |
|---------|----------|
| **Цель** | Научиться писать задания AI как инженер, а не как клиент в комментариях. |
| **Теория** | Структура промпта: контекст → цель → ограничения → критерии → шаги → риски. |
| **Практика** | 10 промптов на одну задачу: от плохого к хорошему (рефактор, багфикс, фича). |
| **Проверка** | AI выдаёт план + уточнения + предложенный diff, а не «вот код наугад». |
| **Ошибки** | «Сделай красиво» без критериев. |
| **ДЗ** | Шаблон `PROMPT_TEMPLATE.md`. |

---

### День 2 — Chat vs Agent/Composer мышление

| Элемент | Описание |
|---------|----------|
| **Цель** | Выбирать правильный режим под задачу. |
| **Теория** | Cursor Agent умеет делать сложные изменения, работать по проекту, запускать команды и править файлы. [→ Cursor Agent](https://cursor.com/docs/agent/overview) |
| **Практика** | • 3 задачи «только объяснить» (Chat)<br>• 3 задачи «изменить 3–5 файлов» (Agent/Composer) |
| **Проверка** | Стажёр умеет формулировать задачу на изменение нескольких файлов и принимать diff осознанно. |
| **Ошибки** | Пытаться «болтать» в чате, когда нужно править код. |
| **ДЗ** | Чек-лист выбора режима. |

---

### День 3 — Контекст: что давать AI, а что не надо

| Элемент | Описание |
|---------|----------|
| **Цель** | Точечно кормить AI нужными файлами. |
| **Теория** | Авто-контекст, ручной контекст, ограничение «всё сразу» (качество падает). [→ Context](https://cursor.com/learn/context) |
| **Практика** | «Одна и та же задача» с разным контекстом: 1) без контекста, 2) с 1 файлом, 3) с 3 файлами, 4) с правилами проекта. |
| **Проверка** | Стажёр умеет собрать минимальный контекст для изменения. |
| **Ошибки** | «Вот весь репозиторий, разбирайся». |
| **ДЗ** | Написать «контекст-пакеты» (наборы файлов) для типовых задач. |

---

### День 4 — Мини-рефактор с безопасной сеткой

| Элемент | Описание |
|---------|----------|
| **Цель** | Делать улучшения без разрушений. |
| **Теория** | Маленькие PR, атомарные изменения, обязательно тест/линтер. |
| **Практика** | Рефактор компонента/модуля + добавить тест/проверку. |
| **Проверка** | PR читается и откатывается без слёз. |
| **Ошибки** | «AI переписал половину проекта, но так надо». |
| **ДЗ** | 1 PR в учебный репозиторий. |

---

### День 5 — Debugging через AI (без гадания)

| Элемент | Описание |
|---------|----------|
| **Цель** | Уметь чинить баги через гипотезы. |
| **Теория** | Цикл: симптомы → гипотезы → проверка → фикс → регресс-тест. |
| **Практика** | 3 бага: UI, API, DB/логика. |
| **Проверка** | Фикс подкреплён логами/тестом, а не «вроде работает». |
| **Ошибки** | «Пофиксил, не знаю почему». |
| **ДЗ** | Документ «Bug report + Fix rationale». |

| **Проверка** | Стажёр находит 5 проблем в заранее «дырявом» коде. |

# Матрица навыков
*Junior → AI-Augmented Developer*

| № | Навык | Уровень |
|---|-------|---------|
| 1 | **Prompting** | От «сделай» → к «сделай по критериям + план + diff + тесты» |
| 2 | **Context control** | Давать минимум нужного, понимать зависимости |
| 3 | **Diff literacy** | Читать изменения как PR, ловить побочные эффекты |
| 4 | **Quality gates** | Линтер/тест/сборка всегда |
| 5 | **Security reflex** | Валидация, секреты, доступы, rate limit |
| 6 | **Docs discipline** | ADR, README, postmortem |
| 7 | **Ownership** | AI предлагает, человек отвечает |

# Критерии допуска стажёра к реальному проекту

Стажёр **допускается**, если он:

- Делает фичи **маленькими PR**, не «одним коммитом на 800 строк»
- Всегда прикладывает **как проверить** + **риски**
- Добавляет тесты или объясняет, почему тест нецелесообразен (редко)
- Умеет сказать: «AI ошибся, вот доказательство»
- Соблюдает правила проекта (`.cursor/rules`) и обновляет их при проблемах

---

# Ошибки, из-за которых нельзя подпускать к production

- Принял diff, не читая
- Убрал/сломал проверки (линтер/тесты) «чтобы прошло»
- Засунул секреты в код/репозиторий
- Сделал «рефактор» вместе с фичей без причины
- Не может объяснить, почему работает фикс
- Не умеет откатить

## Мини-правило курса

> **AI — это ускоритель, не мозг.** Мозг оплачивается отдельно, но у стажёра он должен быть в комплекте.
