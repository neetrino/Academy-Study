# Урок 0.1 — Setup и «контроль реальности»

## Цель урока

Настроить Cursor и репозиторий, понять, что AI делает хорошо и где стабильно врёт (особенно уверенно).

---

## Теория (изучить перед практикой)

| Тема | Кратко |
|------|--------|
| Что AI делает хорошо | Структурированный код, шаблоны, рефакторинг по правилам, генерация тестов по спецификации |
| Где AI стабильно врёт | Точные версии/номера, актуальная документация, «последние изменения» в экосистеме, сложная бизнес-логика без контекста |
| Контроль реальности | Всегда проверять: запуск, линтер, тесты; не доверять «вроде работает» |

---

## Практика — чеклист

### 1. Установка и открытие проекта

- [ ] Установлен **Cursor** (скачать с [cursor.com](https://cursor.com))
- [ ] Открыт **учебный репозиторий** в Cursor
- [ ] Проект **запускается** (команда запуска задокументирована в README или в памяти)
- [ ] (Опционально) Настроен **Obsidian** для чтения уроков — [docs/OBSIDIAN_SETUP.md](../docs/OBSIDIAN_SETUP.md)

### 2. Настройка качества кода (минимум)

- [ ] Настроен **форматтер** (Prettier / встроенный по стеку)
- [ ] Настроен **линтер** (ESLint / аналог по стеку)
- [ ] Настроена **проверка типов** (TypeScript / аналог), если применимо
- [ ] Линтер/форматтер **запускаются** и дают предсказуемый результат

### 3. Документ по работе с AI в команде

- [ ] Создан файл **`docs/AI_WORKFLOW.md`**
- [ ] В документе описано:
  - [ ] как команда работает с AI (когда Chat, когда Agent)
  - [ ] что **запрещено** AI делать без человека (например: коммитить, деплоить, менять критические конфиги)
  - [ ] куда смотреть при конфликтах (правила проекта, ADR)

# Урок 0.2 — Cursor Rules и дисциплина контекста

## Цель урока

Научиться управлять поведением AI через правила. Правила бывают персональные и проектные; **контекст — это валюта**, лишний контекст ухудшает ответы.

---

## Теория (изучить перед практикой)

| Тема | Суть |
|------|------|
| Правила Cursor | Персональные (настройки пользователя) и **проектные** (репозиторий → `.cursor/rules`) |
| Зачем проектные правила | Стиль, стек, запреты, формат PR, требования к тестам — чтобы AI не игнорировал стек и не писал «как попало» |
| Контекст | Чем точечнее контекст (нужные файлы + правила), тем стабильнее результат |

**Ссылка:** [Rules \| Cursor Docs](https://cursor.com/docs/context/rules)

---

## Практика — чеклист

### 1. Создание проектных правил

- [ ] Открыть **Cursor Settings → Rules** (или создать/отредактировать файлы в `.cursor/rules/`)
- [ ] Убедиться, что в проекте есть папка/файл **`.cursor/rules`** (или `rules` в настройках Cursor для этого проекта)

### 2. Содержимое правил (записать в правила)

- [ ] **Стиль кода**: форматирование, нейминг (например: camelCase, компоненты PascalCase)
- [ ] **Стек**: языки, фреймворки, версии (кратко)
- [ ] **Запреты**: что нельзя делать без явного согласования (например: менять зависимости, удалять тесты)
- [ ] **Формат PR**: что должно быть в описании (что сделано, как проверить, риски)
- [ ] **Требования к тестам**: когда тесты обязательны, какой охват ожидается

### 3. Проверка

- [ ] Задать AI задачу в духе «добавь функцию X в проект» и проверить:
  - [ ] AI **учитывает стек** (не предлагает другой фреймворк)
  - [ ] Ответ в **нужном формате** (например, с упоминанием тестов/линтера, если так в правилах)

---

## Критерии проверки (урок засчитан, если)

- [ ] AI **начинает отвечать** с учётом правил (формат, стек)
- [ ] AI **не игнорирует** указанный стек и стиль

---

## Типичные ошибки (избегать)

- Правила на **3 страницы философии** — бесполезно. Нужно **до 15–25 конкретных пунктов** в короткой версии.
- Слишком длинные правила — AI их «забывает» или обрезает. Лучше: короткий набор + при необходимости полный в отдельном файле.

---

## Домашнее задание (Урок 0.2)

- [ ] Сделать **«короткие правила»** (до ~25 строк): только самое критичное (стек, запреты, формат PR)
- [ ] Сделать **«полные правила»** (до ~150 строк): стиль, тесты, примеры, ссылки на ADR/документацию
- [ ] Сохранить в `.cursor/rules/` или в описанном в проекте месте
